# Punk Rock AI Roadmap Refresh - May 7, 2026

This is a state-of-the-project roadmap based on the current repo, docs, code,
recent commit history, and prior delivery notes for `cmvan-keynote`.

## Assumptions

- The talk has already happened. The product surface is now the public
  `punkrockai.com` portal and the May 29 Release Day campaign.
- The repo remains the source of truth for scripts, site code, data contracts,
  deployment notes, and archival handoff. Notion is for sharing and staging.
- Keep the site static, hand-rolled, and visually opinionated. Do not introduce
  a framework unless the current model can no longer carry the work.
- Prefer roadmap items that can be verified with local scripts, route checks,
  live API smoke tests, or concrete publishing artifacts.

## Success Criteria

The next phase is healthy when:

- One command can validate the site contract before pushing.
- Release Day submissions can be tested end-to-end from browser to moderation.
- Docs tell the same story as the code: Vercel-first hosting, shipped portal,
  current blockers, and the actual state of APIs/workers.
- The May 29 campaign has a simple operating rhythm: social posts, moderation,
  gallery publishing, analytics review, and post-deadline archive.
- Post-talk publishing has clear consent, attribution, and source-provenance
  checkpoints before anything public gets expanded.

## Current State

### What is shipped

- `punkrockai.com` is Vercel-first static hosting with `site/` as output,
  `cleanUrls: true`, production deploys from `main`, and static headers in
  `vercel.json`.
- The portal is no longer a minimum companion site. It is a full interactive
  learning surface with Make, Name, Connect, and Ship navigation groups.
- The visual system is strong and documented: punk/zine palette, photo-forward
  heroes, cropped slide assets, typography, decoration primitives, reduced
  motion, and print rules.
- Vercel serverless functions exist for:
  - `api/submissions.js`: Release Day submission queue to Notion with manual
    moderation.
  - `api/subscribe.js`: Beehiiv newsletter signup.
- Cloudflare Worker scaffolds still exist for Pattern Finder and legacy
  submissions, but they are not the primary production path today.
- Data contracts are clear: browser code reads `site/data/*.json`; scripts are
  producers; runtime pages stay static.

### What is still open

- Release Day operations are not fully proven. `site/data/submissions.json`
  is empty, and the moderation-to-gallery publishing loop is still described
  as future cron/manual work.
- `OPEN-QUESTIONS.md` still needs post-talk updates for Adobe involvement and
  recording rights.
- `docs/PROJECT-ROADMAP.md` still contains older "in progress" phase tables
  even though many widgets and APIs have now shipped.
- `SESSION-HANDOFF.md` is useful history but stale in places, including slide
  image notes that conflict with the README's current final-state language.
- Cloudflare fallback docs and `site/_redirects` still include worker
  placeholders. That is fine as fallback, but it should not confuse Vercel
  production operations.
- The repo does not currently have the remembered `npm run eval` gate, CI
  files, or machine-readable `ops/roadmap/features.json` surface.

## Roadmap

### Wave 0 - Make the repo trustworthy again

Target: 1 day.

1. Restore a single verification command.
   - Add `npm run eval`.
   - Include JS syntax checks, JSON parsing, route/data contract checks, and
     build-script dry runs where safe.
   - Keep it pure filesystem/static checks so it works without a local server
     or external credentials.

2. Reconcile docs to current reality.
   - Update `docs/PROJECT-ROADMAP.md` to mark the portal as shipped through
     the May 5-7 work.
   - Move stale talk-day handoff details into an archive section or label them
     clearly as historical.
   - Update deployment docs so Vercel production and Cloudflare fallback are
     impossible to confuse.

3. Create a route and nav contract check.
   - Confirm every nav link has a target file or configured rewrite.
   - Confirm every widget has HTML, JS, CSS, and any required data manifest.
   - Confirm clean URL behavior is covered by Vercel or fallback `_redirects`.

4. Capture current backlog into issues or a machine-readable roadmap.
   - Decide whether to restore `ops/roadmap/features.json`.
   - If restored, make it the source for GitHub/Linear issue text again.
   - If not restored, keep the canonical backlog in this doc plus GitHub issues.

### Wave 1 - Release Day live operations

Target: now through May 29, 2026.

1. Prove the submission path.
   - Test browser form -> `/api/submissions` -> Notion row.
   - Confirm required Vercel env vars: `NOTION_TOKEN`, `NOTION_DB_ID`.
   - Verify invalid URL/name/what validation in production and preview.
   - Write the exact moderation flow: pending -> published -> gallery.

2. Publish the gallery loop.
   - Choose one path: live GET from Notion via API, scheduled static pull into
     `site/data/submissions.json`, or manual export during Release Week.
   - Add an operator checklist for "publish today's submissions".
   - Decide what the empty-state should say before first public submission.

3. Finish public-copy verification.
   - Fill `OPEN-QUESTIONS.md` Q6 Adobe involvement.
   - Fill `OPEN-QUESTIONS.md` Q7 recording rights.
   - Confirm exact CreativeMornings theme/program wording before any public
     claim that might drift from official language.
   - Confirm photo/recording rights for public reuse.

4. Run the campaign rhythm.
   - Maintain a light social cadence through May 29.
   - Track GA4 events/pages that matter: home, recap, Release Day, submission
     starts, submission successes, widget usage.
   - Publish at least one "what people are shipping" update before May 29.

### Wave 2 - Portal hardening and user experience

Target: after Wave 0, parallel with Release Day.

1. Browser QA the real workflows.
   - Home -> Talk -> Three Documents -> Release Day.
   - Release Day form success/failure/fallback.
   - Pattern Finder connected and disconnected states.
   - Mobile nav, desktop nav hover, keyboard navigation, and reduced motion.

2. Performance and accessibility pass.
   - Run Lighthouse on `/`, `/talk`, `/release-day`, `/library`, and one dense
     widget page.
   - Check hero/photo loading, image dimensions, contrast, focus states, form
     labels, and animation behavior.
   - Keep the visual identity intact; this is a polish pass, not a flattening.

3. Data quality pass.
   - Audit `site/data/library.json` for stale scaffolded TBD content.
   - Review `source-material/*` entries that are indexed publicly.
   - Add provenance notes where generated scaffolds and final public copy blur.

4. Widget completion pass.
   - For each widget, classify it as finished, useful-but-static, connected,
     or intentionally fallback-only.
   - Remove or relabel any UI that implies a live backend when it is still a
     copy-paste or local-only path.

### Wave 3 - Connected intelligence and media

Target: after Release Day ops are reliable.

1. Decide Pattern Finder production path.
   - Option A: deploy the Cloudflare Worker and wire `/api/pattern-finder`.
   - Option B: rewrite as a Vercel function for one hosting model.
   - Option C: keep it as copy-paste only and label it honestly.
   - Verify the model name, cost controls, rate limits, cache, and privacy copy
     before enabling live calls.

2. Finish audio and slide media.
   - Generate per-slide cues with ElevenLabs only after voice/rights are clear.
   - Choose local `/public/audio/cues` vs R2/CDN hosting.
   - Re-run `build-audio-cues` and verify `/talk` audio behavior.
   - Complete R2 slide hi-res ingestion only if hi-res lightbox/media value is
     worth the operational complexity.

3. Consolidate backend ownership.
   - Keep Vercel APIs for production or move intentionally to Cloudflare.
   - Avoid maintaining two half-true deployment stories.
   - Update env docs, preflight scripts, and smoke tests to match the chosen
     production model.

### Wave 4 - Post-talk publishing and archive

Target: May 29 onward.

1. Publish the long arc.
   - Banff + Both Hands Full + CMVan + Release Day.
   - Use `docs/PUBLICATION-BIBLIOGRAPHY.md` for citation guardrails.
   - Keep legal/sensitive AI-training claims source-backed and precise.

2. Produce the shareable artifacts.
   - Optional zine/PDF broadsheet from the talk and submissions.
   - Recording clips if rights allow.
   - A polished recap package with photos, slide excerpts, quotes, and links.

3. Archive and sync.
   - Sync final repo state to the kk-bb mirror.
   - Update Notion with commit links and public review handles.
   - Preserve generated-but-gitignored media instructions so a future agent can
     reproduce audio, slides, crops, and galleries.

### Wave 5 - Delivery system and governance

Target: once the product push calms down.

1. Decide whether to restore GitHub/Linear automation.
   - Prior memory says the old delivery model used `ops/roadmap/features.json`,
     generated issue bodies, and `npm run eval`.
   - Current repo does not contain `ops/` or `.github/`.
   - Bring that back only if this project will keep taking feature swarms.

2. Keep issue scope honest.
   - Use `Refs` when a change is useful but narrower than the originating
     issue.
   - Reserve `Closes` for work that actually satisfies the public behavior and
     docs.

3. Keep verification cheap.
   - Every issue should name its local acceptance check.
   - Every backend issue should name its live smoke test.
   - Every publishing issue should name its source/provenance check.

## Guardrails

- Do not make the site generic while hardening it. The punk visual system is a
  product feature, not a skin.
- Do not add backend complexity before Release Day operations are proven.
- Do not publish post-talk claims until rights, attribution, and official
  language are resolved.
- Do not let generated data drift silently. If source markdown changes, rerun
  the producer script and include the generated output in the same change.
